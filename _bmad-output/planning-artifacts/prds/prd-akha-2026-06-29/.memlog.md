- (direction by user) Akha = simple task companion. User loads their own tasks (no preloaded/default tasks); a creature sits patiently in the corner and gives passive supportive messages.
- (direction by user) Target user: needs more than a plain checkbox to-do list, but is NOT interested in an immersive/patronizing companion or an overwhelming list of irrelevant tasks.
- (direction by user) Core values: passive support + encouragement; no overwhelm in onboarding or tasks; no guilt for not using the app; explicitly does NOT try to solve all mental-condition / task-paralysis issues.
- (decision by user) Stakes: polished personal project (medium depth ~5-8 pages). Working mode: coaching path. Source-of-truth inputs: forged-idea.md, market research doc, shipped spec-win-moment-loop.md + docs/akha-spec.md. Brainstorm = background only.
- (note) Research: positioning = 'the companion that grew up' (warm, relatable grown-up, never cutesy). White space = warm companion + serious + genuine no-shame; nobody holds it yet but 'lane is filling'. Sliver = 'outgrown churned adult' from Finch/Habitica.
- (note) Research monetization rec: free core (full companion + win-loop) + paid 'Everywhere' tier (sync/backup/integrations) ~$20-40/yr IDENTICAL on iOS+Android. Don't race to bottom; avoid surprise charges / platform price gaps / gacha. Go/No-Go = conditional GO (niche), NO-GO mass-market.
- (note) Research risks: 'You Are' already claims fair+adult+no-pet (price alone not moat); incumbents could bolt on a character; demand evidence n=2; moat = taste + integrated feeling, not features. Top retention levers: felt first-session win-loop + cross-device continuity.
- (note) Shipped reality: win-moment loop DONE (frontend-only, localStorage per-day counter, restrained, reduced-motion aware). Recurrence engine shipped+tested (37 tests) but app wiring + Base44 migration for recurrence/occurrence_count NOT yet done. Recurrence editor UI not built. Task data already syncs cross-device via Base44.
- (assumption) Spec precedence for PRD: code + integration-notes.md WIN over akha-spec.md. akha-spec.md local-first/IndexedDB/Supabase layer is SUPERSEDED (app is Base44-backed). Calm-overdue redesign = optional edit, not done. Create-your-own-character + Care/goal model = deferred/Ask-First.

- (decision by user) Entry point confirmed: Journey-led. Section order: Vision/bet -> Protagonist -> Journeys -> Features/FRs -> Restraint contract -> Metrics/counter-metrics -> NFRs.
- (direction by user) DIRECTIVE: Finch/competitors are NOT a factor in development. Focus on the user and the user only. Competitor framing was a market-research artifact, not the product compass. PRD must be written user-first.
- (decision by user) The bet: it works because the feeling — satisfaction & contentment (not overwhelm) — comes from visible progress + genuine restraint tied to the user's own real goals. All fluff stripped completely so the felt sense is satisfaction, not overwhelm.
- (decision by user) Why it matters: the user feels accomplished for doing their task AND feels supported/encouraged to keep going.
- (decision by user) Brand non-negotiables (3): (1) no streaks, no shame; (2) never guilts you for absence; (3) never tries to take over your independence / decision-making.
- (decision by user) Protagonist: Ryan, 40, swamped with to-do tasks, overwhelmed with most things in life. Wants a STRUCTURED way of seeing his tasks get completed.
- (insight) KEY INSIGHT (product-shaping): plain to-do lists fail Ryan because the list gets pushed aside and SEEING THE ENTIRE LIST is itself overbearing — the overwhelm takes over. Akha's job is to protect the user from their own backlog, not just store it. The wall-of-tasks view is the enemy.
- (note) Entry moment: morning coffee. Ryan is tired, not looking forward to the day, fearing the long list of work ahead. This is the emotional state Akha meets him in.
- (decision by user) UJ-1 morning session mechanics: (1) open -> creature soft smile + brief encouraging welcome; (2) sees ONLY the first task, with its subtasks collapsed underneath; (3) taps dropdown -> first TWO subtasks appear; (4) completes first subtask -> creature smiles gently + very brief encouragement; (5) completed subtask DISAPPEARS, next subtask flows up, next incomplete subtask opens/becomes more visible.
- (insight) Interaction model from UJ-1: ONE TASK AT A TIME (backlog structurally hidden) + subtasks revealed TWO at a time (progressive disclosure all the way down) + completion = item disappears & next flows up (reveal, not strikethrough) + creature reacts at SUBTASK level.
- (gap) DELTA vs shipped: shipped app shows multi-task Today list and win-moment fires on TASK completion. UJ-1 needs a single-task FOCUSED view + subtask-level creature reactions + flow-reveal. This is new work, not yet built.
- (decision by user) UJ-1 completion arc: all subtasks done -> main task AUTO-ticks complete; completed task DIMS but does NOT fully disappear; progress wheel updates; creature does a little dance; task drops to bottom of the full task list.
- (decision by user) UJ-1 end-of-day: when all tasks complete, the list of dimmed completed tasks SLOWLY disappears from the page (possibly into an archive); creature smiles then lets out a deep slow sigh (contentment/rest payoff — the inverse of the morning dread).
- (insight) Three-tier reaction ladder (proportional, always quiet): subtask=gentle smile+brief line (item vanishes); task=little dance+progress-wheel update (dims & sinks to bottom); day=smile+deep slow sigh (list clears to archive). Subtasks vanish entirely; tasks leave a dimmed visible trace before the day-end sweep.
- (question) OPEN (minor): completed tasks at day-end go 'possibly into an archive' — is there a viewable archive/history, or do they just clear? Resolve when scoping features.
- (decision by user) UJ-2 capture/planning: plan ideally done BEFORE the day, but tasks can be added on the fly as they arrive. Each task gets a PRIORITY 1-5 (5=most important). Priority influences the ORDER tasks are surfaced/completed (the one-at-a-time focus view shows highest priority first).
- (decision by user) UJ-2 tone: capture is a neutral 'get it out of your head' UTILITY moment. The creature just sits and waits — it does NOT engage/react during planning.
- (insight) Resolving design principle: the creature belongs to the DOING, not the planning. And the wall-of-tasks is acceptable during CAPTURE (user is intentionally dumping/organizing) but forbidden during EXECUTION (where overwhelm strikes). Plan freely; do one-at-a-time.
- (decision by user) UJ-2 ordering rule (focus view sort): 1) priority 5->1, 2) due date/time sooner-first, 3) order added earliest-first, 4) manual rearrange override always available.
- (decision by user) UJ-2 subtasks: can be added at task-add OR later when the task comes up; editable anytime; but the attempt should be made to decompose everything UPFRONT (gentle nudge, not hard requirement).
- (question) OPEN (codebase): user wants due date/TIME, but shipped Task model is date-only (YYYY-MM-DD) and recurrence engine is strictly date-only. Adding time-of-day is real new work — resolve in features/NFR pass.
- (assumption) ASSUMPTION (pending confirm): priority always wins ordering; overdue tasks do NOT jump the queue for being late — they sit at their priority, shown calmly (fits no-shame rule). Due date only breaks ties within same priority.
- (decision by user) UJ-3 onboarding (radically minimal): user provides (1) name they want to be called by, (2) colour scheme, (3) creature selection; then a brief warm/encouraging welcome message. NO task tutorial, no setup wizard. NOTE: 'select creature' = pick from PRESETS, not create-your-own (which stays deferred).
